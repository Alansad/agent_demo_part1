Reasoning + Planning（思考链/任务拆解）是什么意思                                                                                                               
─ ─                                      ──────────────────────────────────────────────────────────────────────────────────────────────────────────────
⏺ 先说本质

普通 LLM 调用是：问题 → 直接回答

加了 Reasoning + Planning 是：问题 → 先思考拆解 → 分步执行 → 最终回答

类比前端：就像你接到一个复杂需求，不是直接开始写代码，而是先写技术方案、拆 TODO、再逐步实现。

  ---
Reasoning（思考链 / Chain of Thought）

让模型把"中间思考过程"也输出出来，而不是直接蹦出答案。

❌ 没有 Reasoning（直接回答，容易出错）：

    问：我有 10 个苹果，给了 3 个，又买了 5 个，还剩多少？
    答：12 个


✅ 有 Reasoning（先推理再回答）：

    问：我有 10 个苹果，给了 3 个，又买了 5 个，还剩多少？

    [思考过程]
    初始：10 个
    给出：10 - 3 = 7 个
    购入：7 + 5 = 12 个
    [思考结束]

    答：12 个

看起来答案一样，但复杂问题时有思考过程的准确率显著更高，因为模型在生成每个 token 时都受前面推理内容的约束。

Anthropic 的实现就是 thinking Block，你在上面的结构里已经见过：

{ "type": "thinking", "thinking": "让我逐步分析..." }

  ---
Planning（任务规划）

面对复杂任务，先把它拆解成有序的子任务，再依次执行。

用户说："帮我分析竞品，写一份调研报告"

❌ 没有 Planning：
直接生成一段文字，内容可能不全、逻辑混乱

✅ 有 Planning：
Step 1：确定要调研哪些竞品
Step 2：逐一搜索每个竞品的功能、价格、用户评价
Step 3：对比整理数据
Step 4：生成结构化报告

    → 然后按顺序执行每个 Step，每步可能触发不同工具

  ---
两者结合 = ReAct 模式（Week 3 的核心）

ReAct = Reasoning + Acting，是目前最主流的 Agent 工作模式：

循环执行，直到任务完成：

    ┌─────────────────────────────────────────┐
    │                                         │
    │  Thought（思考）：我现在需要做什么？     │
    │      ↓                                  │
    │  Action（行动）：调用工具 X             │
    │      ↓                                  │
    │  Observation（观察）：工具返回了什么？   │
    │      ↓                                  │
    │  Thought（再思考）：下一步该怎么做？    │
    │      ↓                                  │
    │  ...循环直到任务完成...                  │
    │                                         │
    └─────────────────────────────────────────┘

举个具体例子，用户问「最近 React 有什么新特性，对比 Vue 哪个更适合我的项目」：

Thought：这个问题需要最新信息，我应该先搜索 React 最新版本
Action：search("React latest features 2024")
Observation：搜到了 React 19 的相关内容

Thought：还需要 Vue 的信息
Action：search("Vue 3 latest features 2024")
Observation：搜到了 Vue 3.4 的内容

Thought：用户提到"我的项目"，但我不知道项目情况，需要追问
Action：ask_user("你的项目是什么类型？团队规模多大？")
Observation：用户说是小型后台管理系统，2 人团队

Thought：信息足够了，可以生成对比报告了
Action：（无，直接生成最终答案）
Answer：基于你的情况，推荐 Vue，原因是...

完整 ReAct 长什么样

ReAct 原论文的设计是让模型把思考过程显式输出为文字：

用户：北京和上海哪个更适合出行？

Thought: 我需要分别查询两个城市的天气，然后对比得出结论。
Action: get_weather(city="北京")
Observation: 北京晴天，28°C，微风

Thought: 北京天气不错。现在查上海。
Action: get_weather(city="上海")
Observation: 上海多云，25°C，东南风 3 级

Thought: 两个城市天气都可以，但北京更晴朗，风力更小，更适合户外活动。
Answer: 北京更适合出行，今天晴天微风，上海有风且多云。

每一步都可见、可记录、可调试。

  ---
三种实现方式，复杂度递增

方式一：Prompt 强制输出格式（最简单，Week 3 先学这个）

修改 system prompt，要求模型按格式输出：

system: `
解决问题时必须按以下格式逐步输出：

Thought: <你的推理过程>
Action: <调用的工具名和参数>
Observation: <工具返回的结果>
... 重复直到有答案 ...
Answer: <最终回答>
`

模型会在文字里输出 Thought，但工具调用仍走 Function Calling 机制。

方式二：开启 Thinking 模式（Anthropic 原生支持）

加一个参数，模型的推理过程会以独立 Block 返回：

const stream = client.messages.stream({
model: "claude-opus-4-6",
thinking: { type: "adaptive" },   // ← 加这一行
// ...
});

返回的 content 里会多出 thinking block：

{ "type": "thinking", "thinking": "用户想知道哪个城市更适合出行，我需要先查两个城市的天气..." }
{ "type": "tool_use", "name": "get_weather", ... }

方式三：手动实现 ReAct Loop（最完整，完全掌控每一步）

你的代码显式管理每个阶段：

while not done:
thought = llm.generate_thought(context)    // 强制只输出思考
action  = llm.decide_action(thought)       // 基于思考决定行动
obs     = execute_tool(action)             // 执行并拿到观察
context.append(thought + action + obs)     // 追加进历史
